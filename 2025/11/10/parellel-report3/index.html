四、Problem 1：MPI 并行矩阵–矩阵乘法

4.1 问题描述

给定：

  • 矩阵 $A$：维度 $m \times k$；

  • 矩阵 $B$：维度 $k \times n$；

  • 输出矩阵 $C$：维度 $m \times n$，其中：
    $$
    C_{ij} = \sum_{t=0}^{k-1} A_{it} B_{tj}
    $$

要求：

  1. 从 input1.txt 中读取 $m,k,n$（格式 "8,16,32"）；
  2. 随机初始化 $A,B$；
  3. 在单进程上执行串行矩阵乘法并测时；
  4. 使用 MPI 并行矩阵乘法并测时；
  5. 校验串行结果与 MPI 结果是否一致；
  6. 将 CPU 时间 T1 与 MPI 时间 T2 输出至 output1.txt（格式 "1.23,2.34"）。

实际实现中，同时编写了扩展版本对多组矩阵规模进行基准测试，并输出为 CSV 便于绘图分析。

4.3 MPI 并行算法设计（基于行划分的 Send/Recv）

本实验的 MPI 并行矩阵乘法采用 按行划分矩阵 $A$ + MPI_Send / MPI_Recv 的方式实现，

整体思路如下：

  1. 数据初始化（仅在 rank 0 完成）：

    • rank 0 从 input1.txt 读取矩阵尺寸 m, k, n；
    • 在 rank 0 上分配并随机初始化完整矩阵：
      • 矩阵 $A$ 大小为 $m \times k$；
      • 矩阵 $B$ 大小为 $k \times n$；
    • 其他进程初始不持有完整 $A$，仅等待接收子块。

  2. 尺寸与矩阵 B 分发：

    • 使用 MPI_Bcast 向所有进程广播 m, k, n；
    • 各进程据此分配本地缓冲区；
    • 使用 MPI_Bcast 将完整矩阵 $B$ 从 rank 0 广播到所有进程：
      • 每个进程本地保存一份完整的 $B$，后续仅对 $A$ 做行划分与传输。

  3. 按行划分矩阵 A（Block Row Decomposition）：

    • 令进程总数为 p；
    • 先计算平均行数：srow = m / p；
    • 最后一个进程承担剩余行数：srow_last = m - srow * (p - 1)；
    • 对于rank为 i 的进程：
      • 若 i < p - 1，则处理连续的 srow 行；
      • 若 i == p - 1，则处理最后的 srow_last 行；
    • 实现方式：
      • rank 0 通过多次 MPI_Send 将对应的 $A$ 行块发送给其他进程；
      • 其他进程通过 MPI_Recv 接收自己的子矩阵 A_local。

  4. 本地矩阵乘法计算：

    • 每个进程根据自身拥有的 A_local（维度 local_rows × k）和全局的 B（k × n），调用同一串行内核：

    • 得到对应行块的结果矩阵 C_local（local_rows × n）。

  5. 结果回传与拼接：

    • 非 rank 0 进程使用 MPI_Send 将 C_local 发送回 rank 0；
    • rank 0 使用 MPI_Recv 按进程顺序接收各段结果：
      • 将来自进程 i 的 C_local 写入全局矩阵 $C$ 的对应行区间；
    • 自身负责的前 srow 或 local_rows 行结果则直接写入 $C$ 的起始部分。

  6. 时间测量策略：

    • 串行版本：仅在 rank 0 上用单线程执行 matmul(A, B, C_serial)，测得 $T_{\text{CPU}}$；
    • MPI 版本：
      • 使用 MPI_Barrier 同步所有进程；
      • 以 MPI_Wtime() 包裹 发送 A 子块 + 本地计算 + 回收 C 子块 的完整过程；
      • 使用 MPI_Reduce（MPI_MAX）在 rank 0 汇总各进程耗时，得到 $T_{\text{MPI}}$（由最慢进程决定）。

4.4 实验结果

使用若干规模的方阵（$m = k = n$）进行测试：

（2）结果表格与加速比

m = k = n CPU(ms) MPI(ms) Speedup = CPU / MPI
32 0.105 0.041 2.56×
64 0.855 0.246 3.48×
128 7.048 1.856 3.80×
256 57.611 14.657 3.93×

4.5 分析与讨论

  • 加速效果稳定接近线性：在 4 个进程的前提下，加速比约为 2.56×～3.93×，接近理想 4×。
  • 计算主导时并行更有优势：
    • 尺寸增大时，矩阵乘法的计算复杂度 $O(n^3)$ 快速增长,启动进程的固定开销减少，加速比更接近进程数目（4）；
    • 通信复杂度主要由 $O(n^2)$ 量级的数据分发与收集构成；
    • 因此在中大规模矩阵上，MPI 并行能获得稳定加速。
  • 小规模下通信开销明显：
    • 当矩阵较小时，Bcast / MPI_Send / MPI_Recv 等固定开销相对占优, 导致加速比略低于理想值。

五、Problem 2：Ring-based Allreduce 实验

5.1 问题描述

实现一个自定义的 RING_Allreduce 函数，要求：

  1. 支持操作：SUM 和 MAX；
  2. 使用 点对点通信（MPI_Sendrecv 等） 实现；
  3. 与 MPI_Allreduce 在数值结果上一致；
  4. 对比两者运行时间，并输出到 `output2.txt。

5.2 算法设计

自实现的 RING_Allreduce 基于经典 环形两阶段算法：

  1. 数据分段：
    • 将长度为 n 的数组划分为 P 段（P 为进程数）；
    • 允许不整除，不整除时，前若干段多 1 个元素。
  2. Phase 1：Reduce-Scatter
    • 进行 P - 1 轮；
    • 每轮：
      • 每个进程向右邻居发送一段数据；
      • 从左邻居接收对应段，并根据 op (SUM / MAX) 更新本地结果；
    • 结束后：
      • 每个进程持有其中 一段已完成规约的最终结果。
  3. Phase 2：Allgather
    • 再进行 P - 1 轮；
    • 各进程在环上依次交换已完成规约的段；
    • 最终所有进程都拥有完整长度为 n 的 Allreduce 结果。

整个过程仅使用 point-to-point 通信。

5.3 性能测试与结果

测试（op = sum）：

n op MPI_Allreduce(ms) RING_Allreduce(ms)
512 sum 0.006 0.027
1024 sum 0.009 0.045
4096 sum 0.046 0.095
16384 sum 0.085 0.192

5.4 结果分析

  • 在当前测试环境与消息规模下：
    • 库内置的 MPI_Allreduce 明显优于手写 RING_Allreduce，因为MPI 实现通常采用拓扑感知、流水化、分层树/环等高度优化策略；
    • 不过，在仅进行单次调用的测试中，由于 MPI_Allreduce 首次调用包含内部初始化和算法选择等一次性开销，而自实现的 RING_Allreduce 较为轻量，因此表面上可能出现 “RING_Allreduce 更快” 的现象。在采用多次调用、预热以及取各进程最大耗时的严格基准测试后，可以看到稳态情况下 MPI_Allreduce 的性能优于手写环形实现。

六、实验总结

  • MPI 并行矩阵乘法：
    • 通过合理的数据划分（按行分块）与 Bcast` / `MPI_Send` / `MPI_Recv，数据量较大时在 4 个进程下获得接近 4× 的加速；
    • 表明在计算量足够大时，通信开销可以被摊薄，并行效率较高。
  • 自定义 Ring Allreduce：
    • 正确复现了 All reduce 语义，与 MPI_Allreduce 结果一致，但性能略逊于高度优化的 MPI 实现。
